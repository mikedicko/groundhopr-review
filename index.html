01

Fix this week

● Do these before anything else

These three are different from bugs: they're live risks to money and to the paid product. All are small, targeted fixes. Everything else in this report can wait behind them.

CriticalSecurity · billing

The AI endpoint is an open door to your Anthropic bill

The functions/ai-trip.js function forwards whatever it's sent straight to the paid Anthropic API using your key — with no password, no rate limit, and no checks. It even trusts the caller to pick the model, the length, and the tools. Anyone who finds the URL (it's visible in your public code) can script unlimited requests, run an expensive model, or turn on paid web-search — all billed to you. Open AI proxies get found and abused by bots within days.

▸ Give this to Claude Code
Lock down functions/ai-trip.js. (1) Require a secret header "x-ghpr-key" that
matches a new Netlify env var AI_TRIP_SHARED_SECRET; return 401 if it's missing
or wrong. Update all three fetch() calls to /.netlify/functions/ai-trip in
public/app.jsx to send that header. (2) Ignore any client-supplied "model" and
hard-code "claude-haiku-4-5-20251001". (3) Cap max_tokens server-side at 1500.
(4) Ignore any client-supplied "tools" field entirely. (5) Change the CORS
header from "*" to "https://groundhopr.com". (6) Add per-IP rate limiting of
~20 requests/hour using Netlify Blobs. (7) Delete the console.log lines that
print the API key's presence and length.

Note: a secret embedded in the browser isn't unbreakable — it stops bots, not a determined attacker. The rate limit + model/token caps are what actually bound the cost. That's fine; it's the right first move.

CriticalSecurity · revenue

“Pro” can likely be switched on for free from the browser

Paid access is decided by an is_pro flag on the user's own profiles row, and the app already lets the browser edit that row. Unless the database explicitly forbids it, a user can open dev-tools and set is_pro = true — free Pro forever, bypassing the £3/month. Related: there's no Stripe webhook in the code, so paying customers may not be getting marked Pro automatically at all — a real business bug, not just a security one.

⚑ To confirm: this depends on your Supabase Row-Level Security setup, which can't be seen from the code. Check it today (next card covers how).
▸ Give this to Claude Code
Payment state must never be writable by the browser. In Supabase: add a
BEFORE UPDATE trigger on the "profiles" table that resets is_pro and
trial_expires_at back to their old values unless the request uses the
service_role key. Then create a Stripe webhook as a Netlify function that,
on a successful subscription event, sets is_pro=true using the Supabase
service_role key (which must only live in a Netlify env var, never in the
browser). Point the Stripe subscription at this webhook.
CriticalSecurity · user data

Verify the database lock (Row-Level Security) on every table

Your Supabase key sits in the public code — which is normal and fine, but only because Supabase expects the real protection to live in database rules called Row-Level Security (RLS). Supabase ships new tables with RLS off, and it's easy to miss. If it's off or too loose on any table, then right now anyone on the internet can read and edit every user's saved grounds, trips, tips, comments and friend data. This must be checked before anything else.

▸ Give this to Claude Code
Audit Supabase Row-Level Security. For each table used by the app -- profiles,
pins, pitch_passports, fan_tips, tip_comments, friends -- confirm RLS is ENABLED
and add policies so a logged-in user can only INSERT/UPDATE/DELETE rows that
belong to them (auth.uid() = user_id, or = id for profiles). Keep public SELECT
on pitch_passports by share_token, but restrict its writes to the owner. Give me
the SQL for each policy and tell me which tables currently have RLS turned off.

Bugs users hit right now

● Quick wins — do these next

Features that are shipped and visible but silently don't work. These make the app feel unfinished and are all small fixes.

HighCrash

“Download Trip Plan” crashes every time

The download button in the AI Trip Planner refers to three variables that don't exist anywhere in the code (tripDates, tripDays, tripBudget). Clicking it throws an error and nothing happens — a Pro feature that can't work at all. app.jsx ~3737

▸ Give this to Claude Code
In public/app.jsx around line 3737, the "Download Trip Plan" handler references
undefined variables tripDates, tripDays and tripBudget. The real state in that
component is tripDateFrom, tripDateTo, tripAirport, tripExclude, tripNotes (and
a computed tripDatesDisplay). Fix the handler to use the real variables, drop
tripDays/tripBudget which don't exist, and test that the PDF download works.
HighSilent failure

The 3-day free trial probably never saves

Starting a trial updates the profiles table filtering by user_id, but everywhere else in the code the profile row is keyed by id. If the column is id (as the rest of the code implies) this update matches nothing — the screen flips to “Pro” but the database never records it, so the trial vanishes on the next reload. app.jsx ~5758

▸ Give this to Claude Code
In public/app.jsx, startTrial() updates the profiles table with .eq("user_id",
userId) but every other profiles query uses .eq("id", userId). Confirm the
primary key column name in Supabase, then make startTrial() use the same column
as the rest of the file so trial_expires_at actually persists. Verify the trial
survives a page reload.
MediumBroken promise

Three features look real but do nothing

  • “Attendees” lets you add group emails “so everyone has the trip details” — but nothing is sent or saved; the list disappears on refresh.
  • Contact form shows “Message sent, we reply within 48 hours” but never sends anything anywhere.
  • Matchday Memory theme picker defines 9 ticket themes, but the buttons aren't wired — the ticket is stuck on lime green.
▸ Give this to Claude Code

either wire it up or remove the misleading UI/copy: (1) Pitch Passport

03

What's working well

Worth saying clearly, because the list above is all problems: a lot here is done right, and some of it is better than many funded startups manage.

  • The core idea is strong and focused. “Plan a European away day in one search” is a real need with a clear audience, and the app actually delivers it.
  • The Pitch Passport is a genuinely clever mechanic. Turning trip planning into a 7-step checklist (fixture → flights → hotel → ticket → things to do → pre-match → group) with saved progress and a shareable link is the best part of the product.
  • Search & autocomplete feel professional — type a club and it resolves to stadium + city instantly, backed by a hand-built database of 150+ clubs with coordinates, capacities and official ticket links.
  • The affiliate model is thoughtfully wired across the whole journey (flights, hotels, tickets, activities, rail, gear) with a clean partners bar — and it's monetised sensibly with Pro at £3/month plus a trial and a shirt giveaway.
  • Honest, well-written legal footer — affiliate disclosure, AI-inaccuracy disclaimer, and “we're not a ticket reseller”. Many commercial sites skip this; you didn't.
  • It's a proper installable PWA — offline shell, home-screen install, splash screen, app manifest. That's ambitious for a first build.
  • Real, sensible error/empty/loading states in most places (spinner messages, “be the first to share”, fixtures fallbacks).
  • The brand has personality. “Every ground has a story”, the matey fan voice, the lime-and-black identity, the map of visited grounds — it feels like it's made by someone who loves the thing.
  • Graceful degradation — when the live fixtures API isn't available, it falls back to embedded data instead of breaking.
  • Sensible privacy defaults — a real cookie consent with an “Essential Only” option, not a dark pattern.

05

Rebuild the trip flow as a wizard

● The flagship rework — the biggest single win, once the leaks are fixed

This is the change most likely to move the numbers — completions, affiliate clicks, and shares. Right now the whole trip is one long page of collapsed sections. Turning it into a guided, one-thing-at-a-time flow (the same idea behind the best app onboarding) fits this product unusually well, because a trip is a sequence. Below: why the current version underperforms, a working demo you can click through, the exact playbook, and prompts to build it.

Why the one-page version quietly underperforms

Now · one page, 7 accordions

  • Steps sit collapsed, so most people expand one or two — the hotel, activities and pre-match links (your revenue) are never even seen.
  • “0/7 done” just sits there. Nothing pulls the user forward.
  • Everything visible at once = choice overload; people skim and bounce.
  • The passport almost never gets completed, so the shareable payoff (and your growth loop) rarely fires.
  • Every affiliate link competes for the same glance instead of getting its own moment.

Wizard · one decision per screen

  • Each step is unmissable and full-screen, so every affiliate action becomes the hero of its own screen — far higher click-through.
  • A visible progress bar creates real momentum to finish (the “nearly there” pull).
  • One choice at a time = low effort, high completion.
  • Finishing triggers a celebratory “passport stamped” moment — the natural cue to share.
  • Each partner gets a clean, dedicated pitch at exactly the right time.

The playbook — and why each part beats what's there now

1

One decision per screen

Show a single choice or action at a time. Why better: cuts the choice-overload of the current 7-section page, and each affiliate CTA gets a full screen to itself instead of hiding inside a collapsed row.

2

An always-visible progress bar (“Step 3 of 7”)

Keep progress on screen at every step. Why better: a visible, unfinished bar creates a genuine pull to complete it — the static “0/7 done” badge today does the opposite (it looks like a chore).

3

Front-load a real win

On the first step or two, show an actual flight price or hotel option, not just a link. Why better: people feel value before you ask anything of them, so they commit to the rest of the flow.

4

Pre-fill everything you already know

You already resolve club → stadium → city → dates → fixtures. Carry that into every step so each one is “confirm”, not “fill in”. Why better: removes typing and friction, which is the number-one reason people abandon multi-step flows.

5

Skippable, never hard-gated

Every step has a quiet “Skip for now”. Why better: forcing people through a fixed order kills momentum; letting them skip keeps them moving, and a skipped (un-ticked) step gently invites them back.

6

Save and resume

Persist the current step and choices to the passport row. Why better: a trip gets planned over days, not minutes — people leave and come back. Your pitch_passports table already supports this; it's mostly a UI change.

7

A real completion moment

When they finish, stamp the passport with a little celebration, then immediately offer “Share” and “Add to your grounds map”. Why better: this is the trigger for your organic growth loop — today, because completion is rare, that moment almost never happens.

8

End on a one-page review

Keep today's single-page layout — but as the final summary/booking screen. Why better: the current design is actually great for reviewing everything at a glance; it's just wrong as the building experience. Wizard to build, one page to review and book.

The net effect: you're not adding features — you're re-sequencing the ones you already have so people actually reach them. Same data, same affiliate links, same passport table. The wizard just makes completion the path of least resistance, which lifts both conversion (more affiliate clicks seen) and growth (more finished passports shared).

Booking vs planning: don't send them away mid-flow

The right question to ask before building this: every step ends at a third party (flights on Kiwi, a hotel on Booking, tickets at the club). Forcing someone to actually book each one before they can continue would wreck the flow — that's up to seven separate sites in one sitting, and every jump out is a chance to lose them. The fix is the principle behind every good trip-planning tool: separate deciding from booking.

Book as you go (forced)

  • User must leave to a third-party site at every step just to continue.
  • Each jump risks abandonment; seven sites in a row is exhausting.
  • The plan rarely gets finished, so nothing is saved or shared.

Decide now, book from the summary

  • The wizard only collects decisions — fast, no booking needed to advance.
  • Every booking link opens in a new tab, so the wizard is never lost and progress is saved.
  • A final “Your away day” summary is the one place to book it all — in a batch, or over days.
Recommendation: let people build the whole plan in one quick pass (deciding is easy and keeps momentum), then book from a single summary screen at their own pace. It matches how people actually plan a trip — decide now, book over the week — keeps the flow intact, and you still get every affiliate click; arguably more, because they're gathered in one place the user comes back to. Let keen users book inline (in a new tab) if they want, but never require it to move forward.

After “planned”: the trip needs a home

You've spotted the real gap: “planned” isn't the finish line. A trip has a life of its own after the wizard — things get booked over days, then you travel, then it becomes a memory. So yes, this is a distinct part of the app worth designing: a place the trip lives. Think of it as four stages, each with its own screen and its own job.

1

Plan

The wizard. Decide the shape of the away day. Output: a saved passport.

2

Book & track

The trip dashboard (new). Book each part at your own pace and watch what's done vs still to do. This is the missing screen.

3

Attend

Matchday. A lightweight “today” view — tickets, kickoff, the pre-match spot, your group. Later, not day one.

4

Remember

The trip becomes a pin on your grounds map and a shareable keepsake ticket. Closes the loop back to the map.

Where the share prompt belongs: two moments, not one — when the trip is fully booked (“off to Amsterdam!” bragging rights) and after the visit (the keepsake). Prompting someone to share an empty, unbooked plan is premature. That's exactly why the wizard now hands off to booking, and Share/Add-to-map only appear once the trip is booked.

Stage 2 — Book & track (the dashboard)

How an experienced Apple-school designer would frame it: the finished plan becomes a living object — one screen that answers “what's booked, what's left, and what do I do next?” at a glance. A countdown so it feels real; a simple booked/remaining progress bar; one tap to book each part (opening the partner in a new tab); the ability to tick things off; and Share / Add-to-map surfacing only once it's booked. 06

The backend — strengthen the server layer

Straight answer, Phil: the backend isn't badly written — the three Netlify functions are clean and readable. The real issue is that it's thin and too trusting: decisions that should live on a server (who's paid, vote counts, what the AI is allowed to do) are currently made in the browser, where anyone can change them. You don't need to rewrite anything. You need to add a small, proper server layer for the handful of things that must not be client-controlled. Here's exactly what to build.

CriticalEntitlement · money

Build the missing payment plumbing (Stripe webhook + server-set Pro)

Right now “Pro” is a flag the browser can set, and there's no server code that reacts to a Stripe payment — so it can be faked, and real payers may not be upgraded automatically. The fix is one new server function and one database rule.

▸ Give this to Claude Code
Add real server-side payment handling to the Groundhopr Netlify project.
(1) Create functions/stripe-webhook.js that verifies the Stripe webhook
signature using STRIPE_WEBHOOK_SECRET, and on checkout.session.completed and
customer.subscription.updated sets profiles.is_pro = true for the matching user,
and on customer.subscription.deleted sets it back to false. Use a Supabase
client created with the SUPABASE_SERVICE_ROLE_KEY env var (server-only, never in
the browser). Store the stripe_customer_id on the profiles row so future events
can be matched to the user. (2) In Supabase, add a trigger/policy so is_pro and
trial_expires_at can only be changed by the service_role key, never by the
browser. (3) Register the webhook URL in the Stripe dashboard. Show me the SQL
and the env vars I need to set in Netlify.
HighHardening

Validate and cap every function's input

The functions currently trust whatever they're sent. Add a small guard to each: check the request shape, cap sizes, and whitelist allowed values. (The AI proxy specifics are in Section 01 — this is the same principle applied across all three functions.)

07

The foundation & code quality

● Bigger job — a free weekend, once the above is done

This is the “how it's built” layer. None of it is urgent, but it's the thing that will slow every future change until it's addressed — and one change unlocks most of the fixes.

HighArchitecture

One 6,000-line file, transpiled in the browser

The entire app — ~40 components, all the club data, every database call — lives in one file. It has no build step: the browser downloads a tool called Babel and converts all 6,000 lines to runnable code every single time anyone visits. That's the slow “splash screen” pause you may have noticed. It also means no code is minified, nothing is tested in isolation, and every edit risks breaking something unrelated. This is a totally reasonable way to ship a first app fast — it's just not something to keep building on.

The single highest-leverage change in this whole report. Adding a build step (Vite) fixes the slow load, enables minification, and makes almost every other improvement here far cheaper — splitting the file, tests, linting, safer dependencies. Do the critical fixes first, then do this.
▸ Give this to Claude Code
Introduce a build step without rewriting the app's logic. Set up Vite with
@vitejs/plugin-react. Move app.jsx into src/, then split it into one file per
component under src/components/ and move the big data tables (CLUBS, TRIP_DATA,
etc.) into src/data/. Replace the CDN <script> tags in index.html with npm
dependencies imported normally (react, react-dom, d3, leaflet, @supabase/
supabase-js, html2canvas, jspdf). Update netlify.toml to run "npm run build"
and publish the "dist" folder. Keep behaviour identical; just change how it's
built and loaded. Add a package.json with pinned exact versions.
MediumSafety net

No tests, no linting, and risky dependency loading

  • No linter. A linter would have caught the “Download Trip Plan” crash instantly — it flags undefined variables automatically.
  • No tests. Every change is checked by hand. Even a handful of tests on the pure logic (club search, trip building) would catch regressions.
  • Two libraries float on their latest version (Supabase, D3) with no lock — a library update could silently break login for everyone, at a time you didn't choose.
  • Scripts load from 3 different CDNs with no integrity check — if any were compromised, an attacker could run code on your users. A build step (above) removes this risk entirely.
  • One library (html-to-image) is loaded on every page but never used — remove it.

Housekeeping: commit messages are all “update1”, “meh”, “u1” — when something breaks in a month, history won't help you find when. A one-line habit (“fix trial column bug”) pays for itself the first time you need it. There's also some leftover dead code (an unused old map system) safe to delete.

08

Speed & loading

Mostly downstream of the build-step point above, plus a couple of easy asset wins.

  • Slow first paint from transpiling 6,000 lines in the browser — solved by the Vite build step.
  • Heavy images and video. Several photos are 400–850 KB and the hero background video is ~5 MB, loaded with no compression, modern format, or lazy-loading. Converting to WebP/AVIF and adding loading="lazy" is a big, easy win on mobile data.
  • The offline cache re-downloads the whole app every visit. The service worker tries the network first, so repeat visits don't benefit from the cache unless fully offline. A “show cached instantly, update in background” strategy makes return visits feel instant.
▸ Give this to Claude Code
Two performance passes for Groundhopr. (1) Optimise assets in public/: convert
the large JPEGs and the hero-bg.mp4 to compressed modern formats (WebP/AVIF for
images), add width/height and loading="lazy" to <img> tags, and serve responsibly
sized versions. (2) Change public/sw.js from network-first to stale-while-
revalidate for the app shell so repeat visits load from cache instantly and
